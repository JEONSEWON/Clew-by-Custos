> 이 파일은 Claude Code가 매 세션 자동으로 읽는 **상시 컨텍스트**다. 이 폴더 하나에서
> 개발이 계속 이어지도록, "무엇을 / 왜 / 어떤 규율로" 만드는지를 여기 고정한다.
>
> **이름:** 회사 **Custos**(라틴어 "수호자·파수꾼") · 제품 **Clew**(아리아드네가 테세우스에게
> 준 실타래 — 미로에서 길을 찾게 한 실, 영어 "clue"의 어원). → *에이전트 호출의 미로 속에서
> 길을 추적해 헛도는 낭비를 지켜본다.*

## 1. 한 줄

멀티에이전트 AI 시스템의 트레이스를 받아 **중복 루프·낭비 핸드오프(헛도는 cycle)를
탐지**하고, 거기서 새는 **토큰/비용을 리포트**하는 도구. 궁극적으로는 실시간 차단·감독
(Guardian)으로 확장하지만, 지금은 **사후 탐지 MVP(S0)** 만 만든다.

## 2. 왜 (개념)

멀티에이전트 실패는 대개 "모델이 멍청해서"가 아니라 **에이전트 *사이*의 문제** — 중복
작업, 헛도는 루프, 핸드오프 낭비 — 이고, 이건 곧 *돈*(토큰 낭비)이다. 기존 관측 도구는
단일 에이전트용이라 이 "사이 층"을 못 본다. Clew는 그 낭비를 보이게(→ 결국 막아) 한다.

## 3. 지금 만드는 것 (S0 MVP)

- **입력:** 멀티에이전트 트레이스(우선 한 형식부터 — 예: OpenTelemetry/OpenInference
  spans 또는 LangGraph/LangSmith export).
- **로직(캐스케이드):** 구조적 반복 탐지(후보 cycle) → 의미적 *지역 중복* 확인(반복/형제
  span 출력 유사도) → 낭비 판정.
- **출력:** trace별 "헛도는 cycle 위치 + 중복 정도 + 추정 낭비 토큰/비용" 리포트.
- 자세한 빌드 스펙은 `SPEC.md`.
- **현재 활성 단계 = 1단계** (상세: `SPEC.md` §8).

## 4. 규율 (★ 가장 중요 — 검증 실험의 교훈)

이전 검증 실험(EmergenceTrace)에서 우리는 confounded 신호를 "되는 것처럼" 착각할
뻔했다. 제품에서 같은 실수 = "내가 만들었으니 사람들이 원하는 것"이라 믿는 것. 그래서:

1. **신호는 아직 *우리 데이터*에서 미검증이다.** (v1 실패. 캐스케이드 F1 0.72는 *외부
   연구* 결과지 우리 것 아님.) → **MVP의 첫 임무는 "이 탐지기가 진짜 낭비를 잡나"를
   정직하게 확인하는 것.** 만들었다고 작동한다고 가정하지 말 것.
2. **출시 *전에* 성공/중단 기준을 적어 둔다**(`SPEC.md`에). 결과 보고 기준을 바꾸지 말 것.
3. **검증은 라벨과 분리.** 탐지 로직은 정답 라벨을 참조하지 않는다(누수 금지).
4. **진짜 행동을 측정**(유지율, "진짜 뭔가 잡았다"는 피드백) — 별점·설치수 같은 허영 아님.
5. **"만들었다" ≠ "원한다."** 출시는 신호를 *만드는 것*이지 검증이 아니다. 반응이 검증.
6. 과장 금지(특히 공개 글·README에서 "F1 0.72를 우리가 달성"이라 쓰지 말 것 — 연구 결과임).

## 5. 단계 로드맵 (끝 없음 — 연구↔제품 순환)

S0 낭비탐지(지금) → S1 실시간 차단 → S2 조정 관측(그래프) → S3 수렴/건강성 지능 →
S4 Guardian(자율 감독) → S5 거버넌스 → S∞. **각 단계는 앞 단계가 신호를 준 다음에만.**
